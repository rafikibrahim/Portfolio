2. Firmware Engineering: Real-Time Synchronization

A critical challenge was ensuring frame-perfect synchronization between two separate hardware units without using Wi-Fi (which caused timing conflicts with VGA generation). I implemented a deterministic UART Handshake Protocol. [cite_start]When multiplayer mode is selected, the units lock into a synchronization loop before game logic executes[cite: 2425].

The following C++ snippets represent the actual logic used to handle the multiplayer handshake and controller packet parsing:

File: src/multiplayer_sync.cpp Language: C++ (ESP-IDF) 
// --------------------------------------------------------------------------
// UART Multiplayer Handshake
// Ensures both ESP32 units enter the game loop simultaneously.
// Baud Rate: 115200 (Configured on UART2)
// --------------------------------------------------------------------------

bool PerformHandshake(bool isPlayer1) {
    if (isPlayer1) {
        // Player 1 Initiates
        UART_Send("P1READY");
        int timeout = 5000; 
        
        while (timeout > 0) {
            if (UART_Available()) {
                String response = UART_ReadString();
                if (response == "P2READY") {
                    return true; // Sync Successful
                }
            }
            delay(1);
            timeout--;
        }
    } else {
        // Player 2 Waits for P1
        while (!UART_Available()) {
            // Blocking wait for P1 signal
        }
        String signal = UART_ReadString();
        if (signal == "P1READY") {
            UART_Send("P2READY"); // Acknowledge
            return true;
        }
    }
    return false; // Handshake Failed
}
File: controller/firmware/main.c Language: C (AVR ATmega328P) 
// --------------------------------------------------------------------------
// Wireless Controller Firmware
// Reads 10-bit Analog Joystick and transmits efficient 8-bit packets.
// Communication: Bluetooth Serial (HC-05) @ 9600 baud
// --------------------------------------------------------------------------

void loop() {
    // 1. Read Inputs
    int joyX = analogRead(A0);
    int joyY = analogRead(A1);
    bool btnA = !digitalRead(2); // Active Low
    
    // 2. Construct Packet (Start Byte | X | Y | Buttons | End Byte)
    uint8_t packet[5];
    packet[0] = 0xAA; // Start of Frame
    packet[1] = map(joyX, 0, 1023, 0, 255); // Compress to 8-bit
    packet[2] = map(joyY, 0, 1023, 0, 255);
    packet[3] = (btnA << 0) | (btnB << 1);  // Bitmask buttons
    packet[4] = 0x55; // End of Frame
    
    // 3. Transmit
    Serial.write(packet, 5);
    
    delay(20); // ~50Hz Update Rate
}

4. Engineering Challenges & Solutions

Challenge: The "Everything Everywhere" Crash
[cite_start]

Problem: Running Wi-Fi (for multiplayer), Bluetooth (for controllers), and VGA generation simultaneously caused system-wide failure due to radio resource contention on the 2.4GHz band[cite: 2097].

Solution: I completely removed Wi-Fi from the architecture. I replaced it with a hardwired UART link (GPIO 16/17) between the two ESP32s. [cite_start]This decoupled the multiplayer sync from the radio stack, resolving the crash and lowering power consumption[cite: 2099].

Challenge: Memory Constraints

Problem: The ESP32 ran out of RAM when trying to hold frame buffers for VGA while also managing the Bluetooth stack and Game Logic variables.

Solution: Optimized the video resolution down to 320x200. [cite_start]This reduced the frame buffer size significantly while maintaining a retro aspect ratio compatible with standard VGA monitors[cite: 2677].

Challenge: Audio Latency

Problem: Handling audio directly on the ESP32 introduced jitter into the VGA signal due to interrupt overhead.

Solution: Designed a dedicated Audio PCB with its own ATmega328P. [cite_start]The main console simply sends a brief 27ms GPIO trigger signal, and the dedicated board handles the heavy lifting of MP3 decoding[cite: 2692].

5. Project Validation & Results

The final prototype was subjected to rigorous stress testing. [cite_start]The system demonstrated stable performance with a Bluetooth pairing time of ~4.02 seconds [cite: 2701] [cite_start]and a VGA startup time of ~5.1 seconds[cite: 2723]. [cite_start]Most impressively, the controller's parallel battery configuration achieved an average runtime of 72 hours, far exceeding the initial 6-hour requirement[cite: 2647].
