1. Project Introduction: Reviving Social Gaming

The Concept (What it is):

Project R&F is an innovative dual-screen gaming console designed to bridge the gap between the nostalgic simplicity of retro gaming and modern embedded systems engineering. Unlike traditional consoles that rely on a single shared screen, this system drives two independent VGA monitors simultaneously, providing each player with a dedicated view while maintaining a synchronized game state. The system runs classic titles—Tetris, Pong, and Minesweeper—re-engineered from scratch to support competitive and cooperative multiplayer modes.

The Engineering Challenge (What it is doing):

The system operates as a distributed embedded network rather than a monolithic computer. It offloads tasks across multiple microcontrollers to achieve real-time performance without an operating system:

  • Main Console: Two ESP32-WROVER units communicate via a high-speed, hardwired UART bridge to sync game physics and player states with zero latency.
  • Wireless Control: Custom-built controllers use ATmega328P chips to process analog inputs and transmit data via HC-05 Bluetooth modules.
  • Visuals: Video signals are generated "bare metal" using Direct Memory Access (DMA) and GPIO bit-banging to produce standard VGA signals without a GPU.

The Motivation (Why we did it):

Modern gaming has become increasingly complex, expensive, and isolated, often requiring high-end hardware and internet connections. Our goal was to engineer a low-cost, accessible, and social alternative that prioritizes face-to-face interaction. From a technical perspective, this project served as the ultimate test of our engineering curriculum—requiring the seamless integration of power management, analog circuit design, wireless protocols, and real-time firmware optimization into a consumer-ready product.

My Role as Team Leader & Lead EE:

Leading a team of three Computer Engineers, I was solely responsible for the entire hardware lifecycle. This included the schematic capture and PCB layout of all three system boards (Main Console, Audio Subsystem, and Controllers) using Altium Designer. I engineered the power architecture to support 72-hour runtimes, designed the mixed-signal VGA output stages, and managed the physical fabrication of the enclosures. Additionally, I bridged the gap between hardware and software, writing the low-level drivers for the Bluetooth stack and audio synchronization.

2. System Architecture & Modular Design

The system utilizes a distributed architecture to manage the high resource demands of real-time VGA generation and multiplayer synchronization. Unlike single-processor designs, I separated the system into three distinct PCB subsystems to minimize noise and processing bottlenecks.

Figure 1: High-Level System Architecture connecting ESP32 Master Nodes to Audio and Controller Subsystems.
Main Console Subsystem
  • 2x ESP32-WROVER: Each dedicated to one player's screen and game logic.
  • Inter-Chip Comms: Full-duplex UART at 115200 baud (GPIO 16/17) for zero-latency state synchronization.
  • Video Output: 5-pin Digital VGA interface (3 RGB + H-Sync + V-Sync).
Peripheral Subsystems
  • Audio Board: Dedicated ATmega328P + DFPlayer Mini to offload audio processing from the main game loop.
  • Controllers: Wireless ATmega328P units sending packetized input via HC-05 Bluetooth modules at 9600 baud.

3. Firmware Engineering: Real-Time Synchronization

A critical challenge was ensuring frame-perfect synchronization between two separate hardware units without using Wi-Fi (which caused timing conflicts with VGA generation). I implemented a deterministic UART Handshake Protocol. When multiplayer mode is selected, the units lock into a synchronization loop before game logic executes.

The following C++ snippets represent the actual logic used to handle the multiplayer handshake and controller packet parsing:

File: src/multiplayer_sync.cpp Language: C++ (ESP-IDF) 
// --------------------------------------------------------------------------
// UART Multiplayer Handshake
// Ensures both ESP32 units enter the game loop simultaneously.
// Baud Rate: 115200 (Configured on UART2)
// --------------------------------------------------------------------------

bool PerformHandshake(bool isPlayer1) {
    if (isPlayer1) {
        // Player 1 Initiates
        UART_Send("P1READY");
        int timeout = 5000; 
        
        while (timeout > 0) {
            if (UART_Available()) {
                String response = UART_ReadString();
                if (response == "P2READY") {
                    return true; // Sync Successful
                }
            }
            delay(1);
            timeout--;
        }
    } else {
        // Player 2 Waits for P1
        while (!UART_Available()) {
            // Blocking wait for P1 signal
        }
        String signal = UART_ReadString();
        if (signal == "P1READY") {
            UART_Send("P2READY"); // Acknowledge
            return true;
        }
    }
    return false; // Handshake Failed
}
File: controller/firmware/main.c Language: C (AVR ATmega328P) 
// --------------------------------------------------------------------------
// Wireless Controller Firmware
// Reads 10-bit Analog Joystick and transmits efficient 8-bit packets.
// Communication: Bluetooth Serial (HC-05) @ 9600 baud
// --------------------------------------------------------------------------

void loop() {
    // 1. Read Inputs
    int joyX = analogRead(A0);
    int joyY = analogRead(A1);
    bool btnA = !digitalRead(2); // Active Low
    
    // 2. Construct Packet (Start Byte | X | Y | Buttons | End Byte)
    uint8_t packet[5];
    packet[0] = 0xAA; // Start of Frame
    packet[1] = map(joyX, 0, 1023, 0, 255); // Compress to 8-bit
    packet[2] = map(joyY, 0, 1023, 0, 255);
    packet[3] = (btnA << 0) | (btnB << 1);  // Bitmask buttons
    packet[4] = 0x55; // End of Frame
    
    // 3. Transmit
    Serial.write(packet, 5);
    
    delay(20); // ~50Hz Update Rate
}

5. Engineering Challenges & Solutions

Challenge: The "Everything Everywhere" Crash

Problem: Running Wi-Fi (for multiplayer), Bluetooth (for controllers), and VGA generation simultaneously caused system-wide failure due to radio resource contention on the 2.4GHz band.

Solution: I completely removed Wi-Fi from the architecture. I replaced it with a hardwired UART link (GPIO 16/17) between the two ESP32s. This decoupled the multiplayer sync from the radio stack, resolving the crash and lowering power consumption.

Challenge: Memory Constraints

Problem: The ESP32 ran out of RAM when trying to hold frame buffers for VGA while also managing the Bluetooth stack and Game Logic variables.

Solution: Optimized the video resolution down to 320x200. This reduced the frame buffer size significantly while maintaining a retro aspect ratio compatible with standard VGA monitors.

Challenge: Audio Latency

Problem: Handling audio directly on the ESP32 introduced jitter into the VGA signal due to interrupt overhead.

Solution: Designed a dedicated Audio PCB with its own ATmega328P. The main console simply sends a brief 27ms GPIO trigger signal, and the dedicated board handles the heavy lifting of MP3 decoding.

6. Project Validation & Results

The final prototype was subjected to rigorous stress testing. The system demonstrated stable performance with a Bluetooth pairing time of ~4.02 seconds and a VGA startup time of ~5.1 seconds. Most impressively, the controller's parallel battery configuration achieved an average runtime of 72 hours, far exceeding the initial 6-hour requirement.
