Bark It!: Interactive Cognitive Trainer

[cite_start]

Client: Junior Design Project [cite: 86]

[cite_start]

Location: Pittsburgh, PA [cite: 293]

Year: 2025

[cite_start]Bark It! is an automated, sensor-driven system designed to provide physical and mental enrichment for canines[cite: 2]. [cite_start]By utilizing an ATMega32 microcontroller, the device facilitates a real-time gaming loop that tracks performance and scales difficulty based on the user's success rate[cite: 31, 50].

Design Constraints & Objectives

The development of the trainer was guided by specific functional requirements to ensure accessibility for the animal and reliability for the owner:

    [cite_start]
  • Behavioral Interfacing: Tactile sensors, such as joysticks and large push buttons, were selected to accommodate natural canine movements including tugging and paw-pressing[cite: 9, 11, 40].
    • [cite_start]
  • Cognitive Feedback: The system incorporates multimodal cues, using a 16x2 LCD for visual tracking and an audio module for distinct command tones[cite: 6, 49, 62].
    • [cite_start]
  • Safety-First Architecture: All electronic components and wiring are isolated within a hollow base underneath the enclosure to prevent accidental damage during play[cite: 295, 306].
    • [cite_start]
  • Adaptive Timing: The firmware features a dynamic timing variable that reduces the task completion window as the score increases, maintaining high engagement levels[cite: 50, 199].

System Architecture

[cite_start]

The system randomly cycles through three primary behavioral commands, each requiring a unique sensor interaction[cite: 45]:

    [cite_start]
  • Tug It (Joystick Logic): Detects directional tension when a rope is pulled beyond a calibrated voltage threshold[cite: 9, 46].
    • [cite_start]
  • Dig It (Tactile Input): Monitors a high-visibility red push button for a sequence of three rapid presses to simulate digging behavior[cite: 11, 47].
    • [cite_start]
  • Place It (State-Machine Logic): Monitors the displacement of a ball between two compartments using dual limit switches[cite: 5, 48].

Electronic Design Implementation

[cite_start]I designed a 2-layer PCB optimized for modularity within the enclosure[cite: 93, 98]. [cite_start]To streamline assembly, I implemented 2-pin connectors for all I/O instead of direct soldering[cite: 88]. [cite_start]This design choice allowed for organized wire routing from the central board to sensors distributed throughout the doghouse structure[cite: 89, 149]. [cite_start]The layout includes a keep-out layer to ensure signal integrity and strategic grouping of connectors to minimize interior clutter[cite: 94, 99].

Software Validation & Control Logic

[cite_start]The firmware manages random task generation and continuous sensor polling[cite: 196, 205]. [cite_start]A critical engineering challenge involved the "Place It" command logic[cite: 64]. [cite_start]The code was refined to monitor specific state changes: the first switch must move from active to inactive (ball removed) before the second switch moves from inactive to active (ball placed)[cite: 69, 70]. [cite_start]Furthermore, the system performs continuous monitoring; any incorrect action outside of the active command triggers an immediate game-over state[cite: 337, 338, 340].

Mechanical Fabrication

[cite_start]The enclosure was modeled in SolidWorks and constructed from laser-cut 1/8 inch plywood[cite: 293, 294, 302]. [cite_start]The structure uses sliding joints reinforced with wood glue for high durability[cite: 303, 304, 312]. [cite_start]To ensure sensor reliability, I fabricated custom wooden mounts to position the limit switches at the precise height and angle required for ball activation[cite: 324, 325]. [cite_start]The final assembly features a "porch" for paw-based tasks and a "storage" area for cognitive object movement[cite: 289, 290].

Design Iterations & Future Work

[cite_start]During development, it was determined that ultrasonic sensors were prone to false positives, leading to a pivot to physical limit switches for superior accuracy[cite: 14, 15, 16]. [cite_start]Future iterations will focus on integrating voice-recorded commands to replace simple tones and non-volatile memory to allow high scores to persist after power-cycling[cite: 54, 467, 468].
